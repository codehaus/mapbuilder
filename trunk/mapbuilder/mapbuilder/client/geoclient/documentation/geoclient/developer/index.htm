Welcome

Part of the GeoServer project, GeoClient is a graphical software for online mapping.  GeoClient is designed to work as a "client" to a server operating on the Open GIS Consortium's draft Web Feature Server (WFS) standard.

The GeoServer project's mission is to enable greater geographic interoperability by implementing OpenGIS standards and lowering the barriers to entry for geographic data providers and users.

Written in Scaleable Vector Graphics (SVG) and ECMAscript/JavaScript, GeoClient is designed to run "natively" in a web browser, once SVG support is available.  In the meantime, it uses the Adobe SVG Viewer plugin.  GeoClient is a parallel development to Java-based open source WFS client initiatives like those within the GeoTools project.

For setup instructions, see the user documentation.  This developer documentation provides a more technical introduction to GeoClient for developers.
Model
First off, some general observations on the model of WFS-based web mapping.

There are pretty well two basic models for online mapping. The first and dominant one centres most important functionality at the server. The client (often, a web browser, or an applet or application running from within a web browser) makes requests and the server responds by rendering output (generally in one of the standard web raster formats, e.g., PNG or GIF) that is sent to the client for display. Crucially, data remain on the server; the client loads only enough information to know how to make requests. The pre-eminent open source example of this model is the Minnesota MapServer. Other examples include ESRI's ArcIMS (though ArcIMS can also generate XML output). The OpenGIS Web Map Server specification adopts this first model.

In the second model, major functionality and data manipulation occur on the client side. Instead of doing the full job of rendering a graphic representation, the server instead passes a data stream with both coordinate and attribute information to a client capable of interpreting the information. This may enable a fuller range of GIS-like capabilities on the client side, like:

  • fuller and more complex data manipulation by the user
  • higher-end graphics (i.e., beyond what is possible with server-generated rasters)
  • data export (e.g., for use in vector graphics applications)
  • high-resolution printing
  • input and editing of both feature geometries and attribute data.

The OpenGIS Web Feature Server specification is compatible with this second model.

Clearly, there are also some potential limitations to this second model. Limitations may include:

  • higher demands on the client
  • more expertise required from users
  • larger file transfers/downloads
  • many data sets usable for raster output may not be licensed for vector/data rich output.

So the data rich, vector model used in GeoClient doesn't replace standard web map servers like MapServer. Rather, it aims to provide a rich set of online mapping tools for a subset of web mapping tasks--those where there is a priority for the added functionality of the model and an ability to support the higher demands and other limitations.

Approach
There are three approaches that are typically taken to produce client-side interactive web applications: (1) working with the scripting functionality that is supported "natively" by web browsers (Javascript/ecmascript), (2) creating Java applets that run independently or are embedded in web pages, and (3) designing content that requires proprietary "plug-ins" that run from within a web browser.

The longer-term aim in the GeoClient project is the first approach: full native browser support. However, none of the major web browsers yet offers full support for the XML or, in particular, the SVG standards, so in the meantime GeoClient relies on the Adobe SVG Viewer plug-in, which closely follows the SVG standard and is has support for multiple operating systems and browsers.

The plug-in can be used in one of two different ways: (a) embed the SVG in an HTML page (using an <embed> element), or (b) limit the client to an SVG document. The first option seems intrinsically preferable given that the long-term aim is native browser support, and therefore seamless integration between SVG and HTML/XHTML. HTML integration will be particularly useful for form-based data entry and updating. However:

  • as things stand, the Adobe plug-in is uneven across browsers and operating systems in its support for scripting access to the plug-in; and
  • the plug-in model would necessitate considerable work to ensure cross-browser support.

Because of these limitations, GeoClient makes the following division:

  • pure SVG for the basic end-user functionality;
  • HTML integration for administration and data editing functions (where, e.g., forms are most needed, and browser-specific support is less problematic)--this version not yet available.
Related Projects

Besides the other components of the GeoServer project - most notably, GeoServer, the Java WFS - there are two other open-source initiatives relevant to GeoClient:

  • The GeoTools project is working towards WFS support in their set of online Java mapping tools.
  • The developers of the Minnesota MapServer have announced WFS support for an upcoming release of the MapServer software.  When WFS support is available in MapServer, GeoClient should be adaptable to query MapServer as a WFS.
Client-Server Interactions

GeoClient uses standard capabilities as defined in the WFS specification as well as some "vendor-specific" capabilities (implemented in GeoServerLite).  The vendor-specific capabilities add functionality not available in the standard capabilities.  These are the requests issued by GeoClient and the responses expected from a server:

Request

Server Response
standard capabilities
GetCapabilities Generate a Capabilities document specifying the capabilities of the server.  These include standard capabilities as defined in the WFS specification well as "vendor-specific" capabilities.  Also given are basic data on the project/server plus some settings for each available layer.
DescribeFeatureType Generate an XMLSchema document of the requested layer or layers.
GetFeature Generate Feature Collection document with features, including both attribute and geometry data, in GML format.
vendor-specific capabilities
GetExtendedProject- Descriptor Generate an Extended Project Descriptor document specifying settings for the project beyond those given in the Capabilities document, namely:
  • ScaleFactor, a factor to be used in converting screen pixel coordinates to coordinates in the map projection
  • BBox, the maximum extent served by the project, in coordinates of the project's projection
GetStyledLayer- Descriptor Generate a Styled Layer Descriptor document (as described in the proposed Styled Layer Descriptor specification).
GetExtendedLayer- Descriptor Generate Extended Layer Descriptor document specifying layer settings beyond those given in the Capabilities document, namely: Shape, MinScale, MaxScale, HasAttributes, IDType, NameField, IDField, URL, LegendType, ClassificationField, ClassificationType, ClassesCount, SymbolType, SymbolSize, StrokeWidth, ColorList
(see GeoServerLite developer documentation for descriptions of these fields and their values).
