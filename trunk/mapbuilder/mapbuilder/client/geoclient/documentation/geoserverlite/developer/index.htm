Welcome

Part of the GeoServer project, GeoServerLite is a simple Web Feature Server (WFS--see the Open GIS Consortium's Web Feature Server specification).  

The GeoServer project's mission is to enable greater geographic interoperability by implementing OpenGIS standards and lowering the barriers to entry for geographic data providers and users.

GeoServerLite is written in the PHP scripting language and draws on a MySQL database.

For installation instructions, see the user documentation.

GeoServerLite can be used in two ways:

  • For testing or demonstration, or for use by organizations lacking a dedicated server.  GeoServerLite uses the broadly available MySQL database and PHP scripting language, and so can be implemented quickly in many web hosting environments where the installations required for GeoServer would not be feasible.
  • For use in conjunction with GeoServer or another WFS to provide the "vendor-specific" functionality required by GeoClient.  GeoClient, an online mapping component of the GeoServer project, requires several "vendor-specific" server capabilities.  Although these capabilities are not - yet - available in GeoServer, it is easy enough to use GeoServer (or another WFS) and GeoServerLite in combination, thus providing both the robust data serving of GeoServer and the extended functionality of GeoServerLite.  This is done through coding in the Capabilities document served by the WFS (see detailed instructions below).
The MySQL Database

For storage of geographic data, the most robust and full-featured open source database available is the PostGIS extension to the PostgreSQL database.  Progressing well towards compliance with the OpenGIS Simple Features for SQL specification, PostGIS features native support for several geographic operators as well as strong spatial indexing.  GeoServer (the Java WFS) includes PostGIS support.

However, some very basic geographic data serving and spatial filtering is possible with any database supporting the SQL standard.  GeoServerLite is designed for use with a MySQL database (though it could fairly readily be extended to support PostGIS).

XML Parsing

A Web Feature Server must both parse incoming XML requests and generate XML responses.

There are several major approaches to parsing XML, most of which have been at least sketched in in PHP (e.g., using DOM objects, using XLink format).  GeoServerLite uses the simplest and best-supported XML parsing in PHP: Simple API for XML (SAX).  Rather than reading in the whole XML document at once, the SAX parser takes in each tag or node as it comes and feeds its contents into a designated function.  See PHP documentation for information on how enable and use the SAX PHP extension, which uses the expat library.  Windows-specific note: you may need to copy the expat.dll file to your Windows system path.
Files in the Distribution
  • simple_wfs.php: the primary file and contains the main parsing functions.  This is the file posted to by a WFS client.
  • connect.php: settings for the MySQL database connection.
  • convertgeom.php: handles well-known text parsing into GML.
  • ELP.php: generates Extended Layer Descriptor document.
  • EPD.php: generates Extended Project Descriptor document.
  • SLD.php: generates Styled Layer Descriptor document.
  • XSD.php: generates XMLSchema document.
Customizing Capabilities

The "Capabilities" document generated by a Web Feature Server describes the server, the data it serves, and what functionality is supported.  GeoServerLite can be used "out of the box", but users will probably wish to customize at least the Capabilities.

To do so, open the file capabilities.php, part of the geoclient distribution.

The simplest customization to make is to give the server a name and description.  By default, this section of the Capabilities document reads:

<Service>
<Name>SimpleWebFeatureServer</Name>
<Title>Simple Feature Server</Title>
<Abstract>Open source simple web feature server</Abstract>
<OnlineResource><?=$thePath?></OnlineResource>
</Service>

Just edit the text nodes to give the server a new Name, Title, or Abstract.

The Capabilities document can also be edited to point specific capabilities to their own URL.  To get the vendor-specific funtionality of GeoServerLite - needed for full use of the GeoClient viewer - but also access robust WFS functionality, you can use GeoClient in conjunction with another WFS.  Here are sample instructions, assuming that you are setting GeoServerLite to work with GeoServer:

  • Set up both GeoServerLite and the more robust WFS, GeoServer
  • Choose one of the WFSs as the source of the GetCapabilities document.
  • Edit that WFSs capabilities to point all standard capabilities to GeoServer and all the vendor-specific ones to GeoServerLite.
